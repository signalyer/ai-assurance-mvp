02The three-layer assurance framework

Every control, eval, finding, and gate the platform produces is anchored to one of these three layers.

Layer 1

Governance

Frameworks: NIST AI RMF NIST AI 600-1 (GenAI Profile)

Defines accountability, risk classification, ownership, human oversight, data policies, evidence requirements, and lifecycle governance.

Example controls

  • AI inventory required
  • Business + technical owner required
  • Risk classification required at intake
  • Human approval required for high-risk actions
  • Evidence completeness ≥ 85% before pilot

Example evidence

  • Intake record · ownership record
  • Risk classification output
  • Approval history · exception waivers

Layer 2

Security

Frameworks: OWASP LLM Top 10 OWASP Agentic AI Top 10

Controls prompt injection, sensitive data leakage, unsafe tool use, excessive agency, RAG poisoning, memory poisoning, delegation abuse, and runtime abuse.

Example controls

  • Prompt-injection testing required
  • RAG source quarantine + version pin
  • Tool authorization mandatory
  • Tool allowlisting mandatory
  • No raw PII/NPI/PCI in prompts

Example evidence

  • Garak / PyRIT reports
  • Tool authorization trace
  • DLP output · RAG quarantine log
  • Runtime policy events

Layer 3

Runtime Assurance

Capabilities: DeepEval Ragas Garak PyRIT Langfuse AWS telemetry Release Gates Runtime Monitoring Evidence Repository Remediation Workflows Assurance Model Providers

Proves the system behaves safely before and after deployment.

Example controls

  • Evals required before release
  • Runtime monitoring required
  • Kill switch required per system
  • Full audit logging required
  • Model-as-judge usage logged

Example evidence

  • Eval results · runtime traces
  • CloudTrail · Security Hub findings
  • Macie / GuardDuty events
  • Model Usage Audit records

03AI workload lifecycle — 12 steps

Every governed AI system moves through these 12 steps. Each step creates a typed platform object and has explicit exit criteria.
1
Register AI System
What happens · Business and technical metadata captured at intake.
Owner · Business Owner + Technical Owner
Object · AISystem
Exit · Required intake fields complete.
2
Classify Risk
What happens · Six deterministic rules score inherent risk (LOW/MEDIUM/HIGH/CRITICAL).
Owner · AI Governance
Object · RiskClassification
Exit · Risk level + rules fired recorded for audit.
3
Apply Required Controls
What happens · Platform selects all P0/P1 controls whose applicability predicate matches the system.
Owner · AI Governance + Technical Owner
Object · ControlAssignment
Exit · Control set assigned to system.
4
Upload Architecture and Evidence
What happens · Architecture diagram, IAM/Terraform/Bedrock snapshots, model + prompt + tool versions attached.
Owner · Technical Owner + Cloud Security
Object · Evidence
Exit · Minimum architecture evidence pack present.
5
Run Assessment
What happens · Per-control evaluation, residual-risk score, framework coverage, release recommendation.
Owner · Model Risk Management
Object · Assessment
Exit · Assessment report stored with rule_fired + rationale.
6
Run Eval Suite
What happens · Prompt-injection, groundedness, hallucination, PII-leakage, tool-authorization evals execute.
Owner · MRM + AI Platform
Object · EvalResult, Evidence
Exit · All evals completed and thresholds evaluated.
7
Generate Findings
What happens · Failed controls + failed evals produce typed findings mapped to ≥1 framework.
Owner · AppSec + AI Platform
Object · Finding
Exit · Findings owned + SLA assigned.
8
Execute Release Gates
What happens · 10 deterministic gates (RG-001..RG-010) evaluated against live data.
Owner · AI Governance + AppSec + MRM
Object · GateResult
Exit · Decision: Approved / Conditional Pilot / Hold / Reject.
9
Review Remediation Plan
What happens · Open findings receive remediation tasks + SLAs by severity.
Owner · Technical Owner
Object · RemediationItem
Exit · Remediation plan accepted by AI Governance.
10
Approve / Hold / Reject
What happens · CRO + CISO + AI Governance Board record approval decision.
Owner · CRO + CISO + AI Governance Board
Object · Approval
Exit · Decision logged with conditions + signatures.
11
Monitor Runtime Behavior
What happens · Langfuse + Bedrock Guardrails + CloudTrail + Macie + Security Hub feed runtime events.
Owner · AI Platform + AppSec
Object · RuntimeEvent
Exit · Kill switch + monitoring + audit logging active.
12
Periodically Reassess
What happens · Quarterly (or post-incident) re-run of assessment, evals, and release gates.
Owner · AI Governance + MRM
Object · Assessment (new version)
Exit · Decision refreshed; waivers re-validated.

06Release decision rules

Decisions are deterministic — the rule that fires is part of the record.

Approved Production

  • All mandatory controls pass
  • No P0/P1 findings open
  • Evidence completeness ≥ 95%
  • Runtime monitoring enabled
  • Audit logging enabled

Approved Pilot

  • No P0 blockers
  • Limited user population
  • HITL enabled where required
  • Evidence completeness ≥ 85%

Conditional Pilot

  • P1/P2 issues remain
  • Compensating controls approved
  • Exception expiration date exists
  • Monitoring level increased

Hold

  • Any P0 finding open
  • Failed PII-leakage eval
  • Failed prompt-injection gate
  • Failed tool-authorization gate
  • Missing required evidence

Reject

  • Architecture violates hard policy
  • Autonomous high-risk FS action without HITL
  • Raw restricted data in model context by design
  • Required trust boundary cannot be met

08Assurance Model Providers — governed model-as-judge

The platform may use models to evaluate, summarize, explain, and draft governance artifacts, but provider usage is governed by data policies, trust boundaries, sanitization, and audit logging. Production runtime models are separate from assurance runtime models.

Production Runtime Models

Used by AI systems themselves
  • Invoked by customer-facing AI agents
  • May execute business workflows / move money / open accounts
  • Stricter trust boundaries
  • Bedrock or Local/VPC models preferred for regulated workloads
  • Subject to release gates + runtime monitoring

Assurance Runtime Models

Used by the platform to govern AI
  • Eval judging · hallucination / groundedness grading
  • Findings summarization · release-decision narratives
  • Evidence summarization · executive report drafting
  • Receive sanitized traces, synthetic prompts, redacted evidence, metadata
  • OpenAI / Anthropic allowed only for sanitized data

Routing rules

  • Public / Internal data → any approved provider.
  • Sanitized confidential → OpenAI / Anthropic allowed if policy permits.
  • Raw PII / NPI / PCI / payment / AML / KYC / credit data → Bedrock or Local/VPC only (exception otherwise).
  • Every model-assisted analysis creates a Model Usage Audit event.
  • Blocked provider use must explain why and recommend alternatives + remediation.
AI System→ Sanitization Layer→ Provider Routing Policy→ Assurance Model→ Eval / Finding / Report→ Model Usage Audit

09Roles and responsibilities

Every step of the SOP names an owner. These are the recurring roles.
Business Owner
  • Owns the business use case
  • Accepts operational business risk
  • Participates in approval decisions
Technical Owner
  • Owns the architecture
  • Provides evidence artifacts
  • Remediates technical findings
AI Governance
  • Owns policy + lifecycle governance
  • Approves exception waivers (≤90 days)
  • Chairs release governance reviews
AppSec
  • Validates OWASP controls
  • Reviews prompt injection, RAG, tool, and runtime risks
  • Owns red-team campaigns
Model Risk Management
  • Validates eval methodology + thresholds
  • Reviews hallucination, groundedness, model behavior
  • Independent assurance role
Cloud Security
  • Validates AWS posture
  • Reviews IAM, KMS, CloudTrail, Macie, Security Hub, GuardDuty
  • Owns network boundary controls
Internal Audit
  • Reviews evidence completeness
  • Validates control operation
  • Checks approval lineage end-to-end
AI Platform
  • Operates model providers + eval pipelines
  • Maintains runtime governance + telemetry connectors
  • Provides the platform infrastructure
