§0Overview & reading guide

This document is the built-state companion to docs/architecture/target-architecture.html. The target doc describes what the platform aspires to be; this one describes what has actually shipped through Day 11 of the 12-day production sprint (Sessions 01a, 01b, 02, 03, 04, 05, 06, 07, 08, 09, 10, 11). Phase-2 deferrals are explicitly marked — both visually (dashed stroke + hatch fill) and in the footer legend — so an auditor or new engineer can tell, without reading any prose, which boxes correspond to real code and which are roadmap.

The most important runtime invariant is the decorator chain order — it appears on the second diagram, in declared order:

@policy_gate → @scrub_pii → @guardrails → @trace_llm_call → @evaluate_response

The order is enforced at SDK decoration time (not call time) by sdk/signallayer/order_guard.py; a missing @scrub_pii raises MissingScrubberError before the function ever runs.

§1Six-Layer System Map

The platform is layered, not microserviced — a single FastAPI app on Azure App Service serves all six layers, with role-based views. The split keeps the v2 refactor cheap (Team Portal + Gov Console as separate clients) without paying a microservice tax in v1.

§2Decorator Chain & Fail-Closed Exits

Five decorators, one enforced order, four explicit fail-closed exits. The chain is the single thing every assurance-tagged code path shares — get the order wrong and the SDK refuses to load the module.

§3Data Plane — events.jsonl is the SSOT

Every state change writes to events.jsonl (append-only, SHA-256 chained). The Session-09 projection worker materialises Postgres views from that stream. UIs render the views. Postgres never writes back into events.jsonl — that direction is a drawing error and a code-review fail. Replaying events.jsonl reconstructs every materialised view.

§4Request Lifecycle Sequence

Two middlewares sit outside the decorator chain: HMACAuthMiddleware guards the SDK boundary (/api/sdk/*), and SessionAuthMiddleware guards browser sessions. A single X-Request-Id propagates from the edge through every hop; Session 11 added the regex validator (^[A-Za-z0-9_-]{1,64}$) to refuse malformed client-supplied IDs.

§5Right-to-Forget Cascade Sequence

One POST /api/right-to-forget cascades across four stores in fail-closed order. Each step returns a PurgeResult with a SHA-256 digest of post-state, the whole cascade is captured as one tamper-evident event, and the new HMAC-signed sidecar at data/rtf_completed_index.jsonl closes the integrity gap surfaced in the Session-10 security review.

§6Demo Control Surface (Session 11)

The Demo Control panel triggers six scenarios; each calls the real backend function, never a mock. Auth gates with require_role("demo-operator", "ciso"). Each run emits a demo_scenario_run event to events.jsonl. _RUNS is bounded at 200 entries (LRU eviction) so a long demo session can't leak memory.

§7Frameworks Coverage Surface

Six frameworks ship in v1, each with a YAML catalog under frameworks/, a generator function in pdf_report.py, and a shared rendering base in domain/pdf_pack_base.py (extracted in Session 11). The coverage matrix is computed live by domain/framework_coverage.py; the matrix is the data, the PDFs are the deliverable.

§9Phase-2 deferrals (called out so the demo stays honest)

Bedrock provider — L6 stub. Anthropic + OpenAI ship in v1; Bedrock is a Phase-2 add.
Multi-language SDKs — Python only in v1. Node, Go, Java are roadmap.
Team Portal / Gov Console split — single FastAPI app with role-based views in v1; the v2 split is a refactor, not a rewrite.
Real-time webhooks — agent version updates use polling; webhook delivery is Phase-2.
External blockchain anchor for the audit chain — internal SHA-256 hash chain is sufficient for v1.
Streaming evals — batch eval only in v1. Eval-as-traffic-flows is Phase-2.
Mobile / responsive console — desktop-first. Mobile is Phase-2.
Per-tenant Azure AI Search index — single index for MVP; per-tenant is Phase-2.
Langfuse delete in RTF cascade — flag-gated stub (LANGFUSE_DELETE_ENABLED=false default). Set to true in production once the API integration is validated.
RTF strict-reject mode — Session 11 ships in migration mode (logs + falls back on unsigned sidecar entries). Session 12 flips to strict-reject after rtf_sidecar_unsigned_total stays at 0 for a week.
