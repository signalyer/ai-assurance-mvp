CLAUDE.md
# CLAUDE.md — AI Assurance Platform | aigovern.sandboxhub.co

## Before every task
Read ARCHITECTURE.md before writing any code.
Confirm by stating the current decorator chain order and
the three most recent "in progress" files.

## Code standards
- Full updated files only — never partial functions or snippets
- Type hints on all parameters and return values
- Docstring on every public function
- `from __future__ import annotations` at top of every Python file
- Pydantic v2 for domain models — ConfigDict, not Config class
- Read every existing file before modifying it
- `python -c "import <module>"` must pass before next file

## Security rules (never violate)
- scrubber.tokenise_payload() runs BEFORE tracer.trace_call()
  Langfuse gets scrubbed_prompt — never raw_prompt
- Policy engine errors → default DENY, never ALLOW
- No SaaS guardrails — all self-hosted, no external prompt routing
- No secrets in code — all config via environment variables

## Storage rules
- JSONL only via storage.py _append_jsonl() and _read_jsonl()
- No direct file writes outside storage.py pattern

## File placement
- New root modules: beside tracer.py (scrubber.py, providers.py)
- New domain: domain/<name>.py — follow domain/repository.py
- New API routers: api/<name>.py — mount in dashboard.py
- New UI: static/<name>.html — follow static/runtime.html
- New middleware: middleware/<name>.py
- Policy files: policies/<name>.rego

## When blocked
Stop. State the blocker. Never fake output. Never work around silently.

## End of every session
1. Run /verify — show all output
2. Update ARCHITECTURE.md — move completed items
3. Write next session plan file to docs/plans/
4. List deviations and open issues

## Compound engineering rule
Every mistake I correct → add a new rule to this file immediately.
Label it with the date. This file grows with experience.
Compound engineering Every time Claude Code gets something wrong during a session, add a rule: "NEVER call tracer.trace_call() with raw_prompt — add 2026-05-20". After ten sessions, CLAUDE.md encodes all hard-won lessons automatically.

# DECISIONS.md — Immutable architectural decision log
# Never edit existing entries. Append only.

## 2026-05-20 — Scrubber before Langfuse
Decision: scrubber.tokenise_payload() before tracer.trace_call()
Alternatives: post-hoc Langfuse redaction, Langfuse self-hosted only
Why: Langfuse Cloud is SaaS — even self-hosted must never receive raw PII
     as defence-in-depth. Constraint survives backend swap.
Constrains: all LLM call wrappers follow scrub→trace order permanently

## 2026-05-20 — OPA as primary policy engine
Decision: OPA (Rego) primary, Cedar secondary
Alternatives: Cedar-only, home-grown rules dict, NeMo for policy
Why: OPA most mature, multi-cloud, git-backed, 9500+ tests.
     Home-grown = unmaintainable at scale.
Constrains: policies/ contains *.rego files, evaluated via OPA sidecar

## 2026-05-20 — No SaaS guardrails
Decision: all guardrails self-hosted (LLM Guard, NeMo, Llama Guard 3)
Alternatives: Lakera Guard, Azure Content Safety
Why: SaaS guardrails route prompts externally — direct PII contradiction
Constrains: guardrail stack deployable on Azure ACI with no external calls

## 2026-05-20 — Four-tier agent memory
Decision: T1 in-context / T2 episodic JSONL / T3 RAG / T4 procedural
Alternatives: single vector store for all memory, pure in-context
Why: RAG and episodic memory solve different problems. Conflating them
     creates stale retrieval and unbounded context growth.
Constrains: agent_memory.py manages all four tiers, RAG is read-only corpus

## 2026-05-20 — JSONL storage for MVP
Decision: JSONL flat files via storage.py for de-id vault and episodic memory
Alternatives: Redis, SQLite, Azure CosmosDB
Why: SQLAlchemy already in requirements for future migration path.
     JSONL minimises dependencies for MVP/demo. Swap backend later.
Constrains: vault TTL enforced in application layer, not database layer

assert 'john@example.com' not in scrubbed, 'FAIL: email leaked'
assert '123-45-6789' not in scrubbed, 'FAIL: SSN leaked'

7.4 Ending every session — /handoff

Type /handoff. Claude Code will:

  • Run /verify and show all output
  • Update ARCHITECTURE.md — move completed to Built with date
  • Append new decisions to DECISIONS.md
  • Write the next session plan file to docs/plans/
  • Output the exact opening message for the next session
Why handoff not end The next session plan file is written while this session's context is hot. This is the continuity mechanism. A cold session starting from scratch re-derives everything — a session starting from a plan file starts informed.

Reference
Rules and anti-patterns
What separates a session that works from one that derails.
RuleDoNever do
Interface separation Architecture decisions in Claude Chat. Execution in Claude Code. Ask Claude Code "what should I use?" — it will invent an answer.
Session scope One component per session. Split HIGH-cost sessions into a/b. Build policy engine + guardrails + RAG in one session.
Context management /compact manually at 50%. /handoff at 70%. Wait for auto-compact at 83.5% — it's lossy.
Plan first Always use /plan and approve before execution starts. Let Claude Code start writing code from the opening message.
CLAUDE.md size Keep under 150 lines. HOW to work only. Put architecture decisions in CLAUDE.md — they get ignored past 150 lines.
Compound engineering Every mistake → new CLAUDE.md rule immediately, with date. Correct the mistake and move on without recording it.
Verification Run /verify before every session ends. Show all output. Accept "it should work" without running the acceptance criteria.
Handoff Write next session plan file while current context is hot. Start next session cold and re-explain the architecture from scratch.
Diagram currency Run /diagram after every session that changes ARCHITECTURE.md. Let the diagram drift from the actual system state.
The one-sentence summary Claude Chat is where you think. Plan files are where thinking becomes executable. ARCHITECTURE.md is the single source of truth. CLAUDE.md is the institutional memory. Claude Code sessions are short, scoped, and verified — never long, open-ended, or trusted without a passing /verify.
